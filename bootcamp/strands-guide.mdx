# Part 1: Introduction to TraceCtrl

## What is TraceCtrl?

TraceCtrl is a **security observability platform for agentic AI**. Our tagline captures the workflow:

1. **See** — Visualize your agent's architecture, topology, and risk posture at a glance
2. **Trace** — Trace every message, tool call, and model request through the full execution lifecycle
3. **Ctrl** — Control and harden your agent's configuration with automated scanning and remediation

## Components

TraceCtrl has five components:

| Component | What it does | Technology |
|-----------|-------------|------------|
| **CLI** | `tracectrl scan`, `fix`, `doctor`, `setup` | Python (pip installable) |
| **Scanner** | Security/ops/perf/compliance checks for agent configs | Python |
| **OTel Collector** | Receives spans from agents via OTLP | OpenTelemetry Collector |
| **Engine** | Processes spans, builds topology, serves API | FastAPI + ClickHouse |
| **Dashboard** | Visualizes scans, traces, topology | React + Vite + Cytoscape.js |

**How data flows:**

1. Your agent (or OpenClaw) exports OpenTelemetry spans to the **OTel Collector**
2. The Collector writes spans to **ClickHouse** (column-oriented database optimized for traces)
3. The **Engine** runs a pipeline every 60 seconds — reads new spans, builds agent inventory, discovers topology edges
4. The **Dashboard** queries the Engine API to render trace trees, topology graphs, and scan reports
5. The **CLI Scanner** analyzes OpenClaw config files and uploads findings to the Engine

# Part 2: Start the TraceCtrl Stack

## What You'll Have Running

By the end of this section you will have:

2. The **tracectrl CLI** installed in a Python virtual environment
3. The Dashboard accessible at [http://localhost:3000](http://localhost:3000) — ready to receive spans from the Strands agent we'll build next

## Clone and Install the CLI

```bash
git clone https://github.com/tracectrl/tracectrl.git
cd tracectrl

python3 -m venv .venv
source .venv/bin/activate

pip install tracectrl
```

## Start the Stack

```bash
tracectrl setup
```

The interactive wizard starts the four Docker containers. Alternatively:

```bash
docker compose up -d
```

<Warning>
  **Rancher Desktop users:** `tracectrl setup` detects Rancher automatically. If you're using the containerd engine, it uses `nerdctl compose` instead of `docker compose`.

  **Memory:** Ensure your container runtime has at least 6 GB of memory allocated (Settings → Resources → Memory). ClickHouse is the heaviest container at 2–4 GB RAM.
</Warning>

**FinFlow AI** is our reference Strands application — an enterprise financial operations platform that automates the full invoice lifecycle: ingestion → policy validation → vendor verification → risk screening → payment execution → notification.

## 3.2 How to Trace

The instrumentation is **three lines plus one decorator** — identical across both examples. Open `agents_workflow.py` (research example) or `teachers_assistant.py` (teacher example) and look at the top:

```python
import tracectrl
from tracectrl import tag_agent
from tracectrl.instrumentation.strands import StrandsInstrumentor

tracectrl.configure(
    service_name=os.getenv("TRACECTRL_SERVICE_NAME", "agents-workflow"),
    endpoint=os.getenv("TRACECTRL_ENDPOINT", "http://localhost:4317"),
)
StrandsInstrumentor().instrument()
```

Then for **every Strands `Agent` you create**, one extra line so TraceCtrl can name it in the topology:

```python
researcher_agent = Agent(
    name="Researcher",
    model=model,
    system_prompt="...",
    tools=[http_request, call_analyst],
)
tag_agent(researcher_agent)
```

Three things to notice:

1. **`StrandsInstrumentor().instrument()`** patches the Strands SDK so every agent invocation, LLM call, and tool call emits an OpenTelemetry span. No per-agent boilerplate.
2. **`tag_agent(...)`** attaches the agent's name and role to its spans. Without it, agents show up as generic nodes — `Researcher` would just be "agent #1" on the topology graph.
3. **OTLP gRPC on `:4317`.** This is the OpenTelemetry Collector that ships with the TraceCtrl stack from Part 2. The Engine reads from it, the Dashboard reads from the Engine.

## 3.4 Add Guardrails with the TraceCtrl SDK

Traces and topology tell you **what your agents do**. Guardrails let you **decide whether to let an action happen** — every input or output runs through a `Guardrail` whose verdict is recorded as a span attribute and surfaced in the Guardrails dashboard.

### The SDK Pattern

The pattern from FinFlow's `payment_delegation_guard` (in `tracectrl-demo/finflow/guardrails/payment_delegation.py`):

```python
from tracectrl.guardrails import Guardrail, wrap_agent_with_guardrails
from strands.models.gemini import GeminiModel

# An LLM judge that scores the wrapped agent's output against a rubric.
judge_model = GeminiModel(
    client_args={"api_key": os.getenv("GOOGLE_API_KEY")},
    model_id="gemini-3.1-flash-lite",
)

my_guard = Guardrail(
    name="my_guard",
    description="One-line summary of what this guardrail enforces.",
    judge_prompt="""\
You are reviewing an agent output. Fail (pass=false) if:
  - <rule 1>
  - <rule 2>
Otherwise pass=true. Output: {output}
""",
    judge_llm=judge_model,
    timing="post_output",      # or "pre_input"
    on_violation="log",        # log | block (block is v2)
    severity="high",           # low | medium | high | critical
)

wrap_agent_with_guardrails(my_agent, [my_guard])
```

Three knobs: **what to check** (`judge_prompt`), **when to check** (`timing`), **what to do on violation** (`on_violation`). The SDK handles span attribution, evidence capture, and dashboard registration for you.

### Guardrail Ideas for the Examples

These are concrete starting points — each takes ~20 lines of Python and slots into the existing example with one `wrap_agent_with_guardrails(...)` call.

<Tabs>
  <Tab title="research_workflow_example">
    **`source_reliability_guard`** — post-output on `Writer`. Fail if the report cites no URLs or cites obvious low-quality sources (forums, social media) without caveat.

    **`fact_check_consistency_guard`** — post-output on `Analyst`. Fail if the verdict (true/false/uncertain) contradicts the evidence the Researcher gathered.

    **`http_egress_guard`** — pre-input wrapper on the `http_request` tool. Block (log-mode for now) requests to localhost / private IP ranges / non-allowlisted domains. Catches SSRF-shaped behaviour from prompt-injected research targets.
  </Tab>
  <Tab title="teacher_assistants_workflow_example">
    **`pii_in_query_guard`** — pre-input on `TeachAssist`. Fail if a student's question contains what looks like PII (email, phone, full name + address) — common when kids paste from somewhere else.

    **`code_execution_safety_guard`** — post-output on `ComputerScienceExpert`. Fail if the generated code contains `os.system`, `subprocess`, `eval`, `exec`, or `rm -rf` patterns. The agent is supposed to teach, not run shell.

    **`scope_guard`** — post-output on every specialist. Fail if `MathWizard` answers an English question, etc. Catches mis-routing and prompt-injection routing hijacks.
  </Tab>
</Tabs>

<Info>
  **Should these live in the examples repo?** Yes — recommendation is a `guardrails/` subfolder in each example mirroring FinFlow's layout (`guardrails/__init__.py`, one file per guard), wired in via `wrap_agent_with_guardrails(...)` in the entry script. That keeps the examples runnable end-to-end with guardrails on or off via a single env var. Flagging this so the examples repo gets that structure before the workshop.
</Info>

# Part 4: GenAI Protector Plus

The SDK guardrails in 3.4 are great for **app-specific** policy (payment caps, scope checks, custom rubrics). For **generic LLM-firewall protections** — prompt injection, PII, content moderation, system-prompt leakage, vector-similarity to known attacks — you don't want to roll your own. cloudsineAI's **GenAI Protector Plus** is exactly that: an external LLM firewall you point TraceCtrl at, and every agent input/output is screened by the same managed guardrails used across your whole org.

The flow:

1. **In Protector Plus:** configure the guardrails you want (Keyword, Shield Prompt LLM, PII, etc.).
2. **In TraceCtrl Settings:** paste the Protector Plus endpoint + API key, tick which guardrails to enable.
3. **Run your agents:** TraceCtrl's SDK now calls Protector Plus on every `check_input` / `check_output` — violations show up alongside your SDK guardrails on the same dashboard.

## 4.1 Configure Guardrails in Protector Plus

In your Protector Plus instance (the cloudsineAI dashboard), enable the guardrails you want. The two we'll use for the FinFlow demo:

**Standard Protection → Keyword.** Exact keyword/phrase blocklist. We add a rule called **"CFO Pre Approved"** with the keyword `CFO-OVERRIDE` — this directly targets the hidden injection text from the malicious PDF.

<Frame>

</Frame>

**Shield Prompt → LLM.** LLM-as-judge for prompt injection. Provider is Ollama (local inference), model `llama3.2:3b`. Faster and cheaper than calling Bedrock/Anthropic for every check; runs on the same machine as the workshop.

After configuring, save and copy two values from the Protector Plus admin:

- **Endpoint URL** (base URL, no trailing slash — e.g. `https://genaifw-sales.example.com`)
- **API key** (domain-scoped, generated under the Protector Plus dashboard)

## 4.2 Connect Protector Plus to TraceCtrl

In the TraceCtrl UI, go to **Settings → TraceCtrl Guards** ([http://localhost:3000/settings](http://localhost:3000/settings)).

Paste the endpoint URL and API key. Tick the guardrails you enabled in Protector Plus — only ticked ones get called from the SDK. For the FinFlow demo, the minimum is **LLM Judge** + **Keyword**.

<Frame>
  <img src="/images/tracectrl-guards-settings.png" alt="TraceCtrl Settings → TraceCtrl Guards page: Endpoint URL field, API Key field (redacted after save), and Enabled guardrails checklist (LLM Judge, PII Detection, Content Moderation, Vector Similarity, System Prompt Protection, Keyword). LLM Judge and Keyword are checked." />
</Frame>

Hit **Save**. The API key field redacts after save — re-enter the full key any time you need to change it.

<Info>
  **What gets called when.** Once enabled, every `check_input` and `check_output` from the TraceCtrl SDK hits Protector Plus for the ticked guardrails. The guardrails show up in TraceCtrl with `protector_plus.` prefixes (e.g. `protector_plus.keyword`, `protector_plus.llm`) so you can tell at a glance whether a violation came from an SDK-defined guard or from Protector Plus.
</Info>

# Bonus: Scan an OpenClaw Workspace

<Info>
  **Fully optional, OpenClaw users only.** Skip this section if you don't already have an OpenClaw workspace running. The OpenClaw attack walkthrough and hardening flow are demonstrated **in person** during the workshop — this section just covers the scan step so you can follow along on your own setup.
</Info>

[OpenClaw](https://docs.openclaw.ai) is a self-hosted AI agent gateway (LLM ↔ messaging channels, with config-driven tool access and session management). TraceCtrl ships a static scanner for OpenClaw configs that runs 30+ security/ops/compliance checks against `openclaw.json` and uploads the findings to the dashboard.

## Enable Diagnostics

If your OpenClaw is already running, add the OTel exporter to `~/.openclaw/openclaw.json` so runtime spans flow to TraceCtrl alongside the scan:

```json
{
  "plugins": { "allow": ["diagnostics-otel"] },
  "diagnostics": {
    "enabled": true,
    "otel": {
      "enabled": true,
      "endpoint": "http://localhost:4318",
      "protocol": "http/protobuf",
      "serviceName": "openclaw-gateway",
      "traces": true
    }
  }
}
```

```bash
openclaw gateway restart
```

## Run the Scan

```bash
# Get your absolute workspace path (run as your normal user, not sudo)
openclaw configure        # first line of output prints the path

# Then scan
tracectrl scan /Users/you/.openclaw/
```

<Warning>
  Don't use `~/.openclaw/` if you're scanning from a `sudo` shell — `~` resolves to `/root/` under elevation. Use the absolute path from `openclaw configure`.
</Warning>

The scanner reads `openclaw.json`, runs the check suite, and uploads results. Terminal output gives you the severity counts and a findings table; the dashboard gives you the full picture.

## View the Scan Report

Open [http://localhost:3000/scan](http://localhost:3000/scan):

3. **Rescan button** (top-right of the findings header) — re-runs the scan in place after you've edited the config. No CLI round-trip needed.
